# Handoff: ntd-compliance-engineer → ingestion-engineer, data-engineer — Slice 2: UPT via TIDES passenger events

## Context
Slice 2 (ADR-0009's successor) delivers Unlinked Passenger Trips. No public event-level APC dataset exists (the TIDES `/samples` directory is template-only, checked 2026-07-10), so passenger events are **simulated — in TIDES format** (`passenger_events` table, TIDES-transit/TIDES spec), which makes the connector a real TIDES connector (ADR-0003 dual status: input adapter AND alignment target) and the simulator's output valid TIDES data. Simulated data must be *permanently distinguishable* in provenance.

## Verified regulatory basis (2026 NTD Policy Manual, quoted 2026-07-10; docs/reference/)
- **UPT definition** (p. 143): "Unlinked Passenger Trips (UPT) are the number of boardings on public transportation vehicles during the fiscal year. Transit agencies must count passengers each time they board vehicles, no matter how many vehicles they use to travel from their origin to their destination. If a transit vehicle changes routes while passengers are onboard (interlining), transit agencies should not recount the passengers. Employees or contractors on transit agency business are not passengers."
- **100% counts / missing-trip rule** (p. 146): "If these vehicle trips are 2 percent or less of the total, transit agencies should factor up the data to account for the missing trips. However, if the vehicle trips with missing data exceed 2 percent of total trips, agencies must have a qualified statistician approve the factoring method." — a REAL FTA threshold (replaces engineering placeholders for this calc).
- **APC validation examples** (p. 151): "agencies may flag trips or blocks where the difference between boardings and alightings is greater than 10 percent, or trips where the passenger load drops below zero."
- **APC certification** (pp. 147–148): FTA approval required; benchmarking ±5% vs manual counts; discard rate < 50%; next benchmarking year RY 2028. (Recorded for context; certification is an agency workflow, not calc logic.)
- **Sampling floor** (p. 149): minimum 95% confidence, ±10% precision — future sampling-path parameters, not used by this slice.

### Schema contract additions (extends handoff 0001; explicit response, not silent)
- **Topic** `raw.tides.passenger_events` (contracts/topics.v0.md): one TIDES `passenger_events` CSV file per message, payload_encoding `object_ref` (same pattern as `raw.gtfs_static.feed`). Producer: `headway-tides`.
- **`canonical.passenger_events`** (migration 0012) — TIDES vocabulary adopted per ADR-0003: `event_timestamp TIMESTAMPTZ NOT NULL` (hypertable time), `service_date DATE NOT NULL`, `passenger_event_id TEXT NOT NULL`, `vehicle_id TEXT NOT NULL`, `trip_id TEXT` (from `trip_id_performed`), `trip_stop_sequence INTEGER`, `event_type TEXT NOT NULL`, `event_count INTEGER` (NULL preserved as NULL — never coalesced), `source TEXT NOT NULL` (envelope source), `source_record_id TEXT NOT NULL REFERENCES raw.records`, unique `(passenger_event_id, event_timestamp, source_record_id)`.
- **Simulated-data rule (binding):** simulator output is ingested with envelope `source = "tides_simulated"` (never `"tides"`). The source column flows to `canonical.passenger_events.source`; any calc consuming simulated records MUST emit an info finding `simulated_source_data` and record the source mix in its detail JSONB. A certifiable figure containing simulated records is a contradiction the DQ trail must make visible.
